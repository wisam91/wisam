# Contributing to Sayr v3

Thank you for your interest in contributing to Sayr! We welcome contributions from everyone.

## Code of Conduct

Please be respectful and inclusive in all interactions.

## Getting Started

1. Fork the repository
2. Create a feature branch: `git checkout -b feature/your-feature`
3. Make your changes
4. Run tests and analysis
5. Submit a pull request

## Coding Standards

- Follow Dart style guide
- Use meaningful variable and function names
- Add comments for complex logic
- Write tests for new features
- Run `melos run analyze` before committing

## Pull Request Process

1. Update documentation if needed
2. Add tests for new functionality
3. Ensure all tests pass
4. Provide clear PR description
5. Request review from maintainers

## Reporting Issues

When reporting bugs:
- Provide clear description
- Include steps to reproduce
- Attach error logs/screenshots
- Specify your environment (Flutter version, device, etc.)

## Commit Messages

Use clear, descriptive commit messages:
```
feat: add GPS tracking feature
fix: resolve login authentication issue
docs: update README with new instructions
test: add unit tests for auth service
refactor: reorganize data layer structure
```

## Branch Naming

- `feature/feature-name` - New features
- `fix/issue-description` - Bug fixes
- `docs/documentation-topic` - Documentation updates
- `refactor/component-name` - Code refactoring

## Questions?

Feel free to open an issue or ask in discussions!
